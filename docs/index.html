01 The Problem

Most organisations treat database backups as an afterthought — relying on manual exports, shared drives with no versioning, or assuming the hosting provider handles everything. When something goes wrong, they discover too late that no reliable, restorable backup exists.

Common failure scenarios: A developer runs a bad migration and corrupts the schema. An admin accidentally deletes the wrong table. A ransomware attack encrypts the primary database. The hosting provider experiences an unrecoverable storage failure. In all four cases, if no automated off-site backup exists, the data is gone permanently.

The solution is not expensive — it is architectural. With the right tooling, a fully automated, cryptographically secured backup pipeline can be built and operated at zero cost, running silently every month without any human intervention.

02 Proposed Architecture

The pipeline is codified entirely in a single YAML workflow file and runs on GitHub Actions — a free, enterprise-grade CI/CD platform used by thousands of organisations globally. No dedicated backup server is required. No paid services are needed.

Execution sequence

1

Scheduler triggers at midnight on the 1st of every month

GitHub's cron system fires the workflow automatically. It can also be triggered manually by any repository admin at any time — useful for pre-migration backups or on-demand snapshots.

2

Temporary Ubuntu runner is provisioned

GitHub spins up a clean, isolated Linux container for the job. It is destroyed immediately after the run completes. No persistent infrastructure to patch, monitor, or pay for.

3

pg_dump connects to the production database and exports everything

PostgreSQL's official native dump utility exports the complete database — all tables, all schemas, all data, all relationships — as a self-contained SQL file that can fully reconstruct the database on any PostgreSQL instance from scratch.

4

Output is compressed immediately in the same pipe

The SQL stream is piped directly through gzip before touching disk. A typical production database compresses to a fraction of its raw size, making storage and transfer trivially fast and cheap.

5

Backup is uploaded to two independent destinations simultaneously

Redundant off-site storage eliminates the single point of failure. If either destination experiences downtime on backup day, the other still captures the data. Both uploads happen within the same workflow run.

03 Storage Destinations

Two independent, geographically separate platforms receive every backup. Neither is under the control of the organisation's primary infrastructure, meaning a compromise of the production environment does not affect the backups.

Why two destinations? Regulatory best practice and the 3-2-1 backup rule both recommend at least two copies stored in at least two different locations. A single backup that lives on the same platform as the production data offers very limited protection.

04 Security Design

No credentials are stored in any codebase or configuration file. All sensitive values are stored as encrypted GitHub Secrets — encrypted at rest using AES-256 — and injected into the workflow's runtime environment only at the moment of execution. GitHub automatically masks these values in all log output, making them impossible to leak through CI logs.

  • Database credentials — Connection string stored as an encrypted secret. Never appears in source code or workflow logs. Rotatable without changing any code.
  • Backup files encrypted in transit — All transfers use TLS. GitHub's API and Telegram's Bot API both enforce HTTPS. The backup is never sent over plain HTTP at any point in the pipeline.
  • Access strictly limited — The Telegram channel is private. GitHub Releases are scoped to repository collaborators only. Neither destination is publicly discoverable or indexed.
  • Full audit trail — Every workflow run is logged with timestamps, exit codes, and output. Any failure automatically triggers an email notification to repository owners. Nothing runs silently without a record.
  • Isolated execution environment — Each run uses a fresh, ephemeral container that is destroyed immediately after. No persistent attack surface. No residual data from previous runs.

05 Disaster Recovery Procedure

In the event of data loss, a full database restoration can be completed by any authorised team member in under 5 minutes using only a terminal and the backup file. No special tools, no vendor support calls, no waiting.

⚠️

Before restoring: If the database still partially exists and contains any recoverable data, export it first. The restore process overwrites the current database state entirely.

Download the backup file

From GitHub: repository → Releases tab → download the most recent .sql.gz file. From Telegram: open the backup channel → download the most recent file attachment. Both contain identical data.

Decompress the archive

terminal bash
# Extract the .sql file from the compressed archive
gunzip database_backup_2026_04_01.sql.gz

Restore to the database

terminal bash
# Replay all SQL statements — fully reconstructs the database
psql "postgresql://[USER]:[PASSWORD]@[HOST]:5432/[DATABASE]" < database_backup_2026_04_01.sql

Verify the restoration

Connect to the database and confirm that all tables, records, and relationships are intact. The application should resume normal operation immediately — no further configuration required.

✅

What is restored: The complete database — every table, every row, all schema definitions, all indexes, all relationships, all stored procedures — exactly as it existed at the moment the backup was taken. Full fidelity. Nothing is missing.

06 Cost & Maintenance

This architecture operates at zero recurring cost on the free tiers of both platforms. GitHub Actions provides 2,000 free minutes per month — a monthly database backup consumes approximately 2 minutes. Telegram provides unlimited file storage at no cost.

Total monthly cost: ₦0. No cloud storage fees. No dedicated server. No third-party backup SaaS subscription. The only requirement is a GitHub account and a Telegram account — both free. This architecture scales from a small team database to a large enterprise database with zero change in cost or complexity.

Maintenance overhead is effectively zero. The workflow is self-contained in a single versioned file. Updates to the database host or credentials require only updating the corresponding GitHub Secret — no code changes, no redeployment.
